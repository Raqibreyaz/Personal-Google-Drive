Architecture Overview

Upload and download flows are designed to avoid the backend becoming a bandwidth bottleneck. Files go directly between browser and S3 via presigned URLs.

Upload flow
Browser → request presigned URL → Backend → presigned URL → Browser → PUT directly → S3
Download / streaming flow
Browser → short-lived presigned URL → CloudFront → S3
Subscription / billing flow
Browser → Razorpay → webhook → Backend → update → MongoDB

File Management Features

Path from Root in Details Popup
Directory

Shows the full hierarchical path of any file or directory from the root, e.g. /projects/2024/assets.

Implementation: Every directory stores an array of ancestor ObjectIds (path field) representing the chain from root to itself. This enables O(1) path reconstruction without recursive queries.
Recursive Directory Stats
Directory

Shows total count of files and subdirectories a folder contains at any depth, computed on-demand when the user opens the details panel.

Implementation: Uses the path field — counts all documents whose path array contains the current directory's ID. This means every descendant at any depth is captured in a single indexed query, avoiding recursive traversal.
Bulk Delete (Files & Folders)
Destructive

Select multiple files and directories and delete them all in one operation, including all nested contents.

Implementation: Frontend sends selected file IDs and directory IDs as separate arrays. Backend deletes files directly from S3 and MongoDB, and for directories deletes all descendants by matching the path field — fully recursive with a single query pattern.

S3 Integration

Direct-to-S3 Upload via Presigned URL
S3

Files upload directly from the browser to S3, bypassing the backend entirely — zero bandwidth cost on the server.

Security: Presigned URL is generated with ContentType and ContentLength conditions enforced. This prevents the user from uploading a different file type or a larger file than what was requested — S3 will reject the PUT if conditions don't match.
Streaming Files Directly from S3
S3 + CloudFront

File downloads and previews are served via CloudFront-backed presigned URLs — frontend talks directly to CDN, never through the backend.

Why not route through backend? A frontend → backend → S3 chain doubles bandwidth costs, adds latency, and creates a scaling bottleneck. Short-lived presigned URLs (minutes TTL) let CloudFront serve requests safely with edge caching. Cache is invalidated on file deletion to prevent stale delivery.
S3 Intelligent Tiering
Cost Optimization

Objects are automatically moved between S3 storage classes based on access patterns, minimizing cost without any manual lifecycle rules.

Problem solved: Standard S3 tier is expensive for infrequently accessed files. Intelligent Tiering monitors read/write patterns and transparently shifts objects to Infrequent Access or Archive tiers — saving significant cost for cold user data.
File Size Limit via Streams
Upload Security

Enforces file size limits at the chunk level during streaming, not after the entire file is received.

Multer limitation: Multer only checks size after buffering the whole file — meaning an oversized upload wastes memory and bandwidth before rejection. Custom stream-based logic reads chunks and aborts the upload the moment accumulated size exceeds the limit.

Upgrade / Downgrade Plan
Razorpay
Constraint: Razorpay only allows plan updates when subscription is in authenticated or active state, and only for card payment method (not UPI). UPI users see a disabled upgrade button with an explanatory tip.

Downgrade guard: If the user's consumed storage exceeds the target plan's limit, the request is blocked and the user is told exactly how many bytes to free up.
Cancellation with Explicit Warning
UX Safety
Problem: Users don't know whether cancelling will result in a refund. A modal shows an explicit warning before cancellation that Razorpay does not issue refunds automatically. Two modes are supported: scheduled cancellation (cancels at billing cycle end, no service interruption) and immediate cancellation (instant demotion to free plan limits).
Grace Period & Expiry Cron Job
Cron
Grace period is the window between subscription cancellation/halt and free-plan demotion — users retain paid storage briefly. A cron job runs two tasks: (1) demotes expired in_grace/cancelled users to maxStorageBytes = 100MB, (2) emails users whose grace period expires within the next 24 hours.
Upload Blocking by Subscription State
Access Control
File uploads are blocked for subscriptions in paused, in_grace, and past_due states. Edge cases handled: cancelled with cancelAtPeriodEnd=true (still active) and awaiting_activation are permitted. Re-subscription is allowed when cancelAtPeriodEnd != true OR graceEndsAt <= Date.now().
⚡

Webhook Event Handling

Webhooks are the source of truth for all subscription state changes. Events are stored in MongoDB on arrival — this enables idempotency checks (skip already-processed events), failure recovery (reprocess failed events), and full audit history.

subscription.activated Updates maxStorageBytes and stores subscriptionId on the user.
subscription.charged Updates current_start and current_end from webhook payload — avoids manual time calculation bugs.
subscription.pending Payment failed and is being retried. Sets status to past_due and blocks file uploads.
subscription.halted All retry attempts exhausted. Sets status to in_grace, enters grace period window, blocks uploads.
subscription.paused Subscription manually paused. Blocks file uploads until resumed.
subscription.resumed Subscription reactivated. Restores upload access.
subscription.updated Plan upgraded or downgraded. Only processed when status is active — other statuses ignored to prevent partial state.
subscription.cancelled Handles both scheduled end-of-cycle cancellation and immediate cancellation with instant free-plan demotion.

Engineering Decisions & Fixes

Issue
Multer checks file size only after receiving the full file into memory — large uploads waste memory and bandwidth before rejection.
Fix
Replaced Multer with a custom stream-based upload handler that checks accumulated chunk size in real-time and aborts the stream the moment the limit is exceeded.
Issue
App Password-based Gmail authentication is insecure — credentials are static and can be leaked.
Fix
Migrated to OAuth 2.0 via Google Cloud Console. Created a project, obtained client_id + client_secret, and generated a refresh token using https://developers.google.com/oauthplayground as the redirect URI. Tokens rotate automatically.
Issue
CloudFront serves stale cached files even after the underlying S3 object is deleted.
Fix
On every file deletion, a CloudFront cache invalidation is triggered for the specific object path. This ensures the CDN serves updated (404) responses immediately rather than serving deleted content from cache.
Issue
Razorpay checkout script loaded on every page adds unnecessary load time to the entire app.
Fix
Script is loaded dynamically only when the user navigates to the plans page. A unique id attribute prevents duplicate injection if the user revisits the page without a full reload.
Issue
Razorpay does not allow plan upgrades/downgrades for subscriptions paid via UPI.
Fix
Plan update UI is disabled for UPI payment methods. A proactive tip at subscription time advises users to choose card payment if they anticipate needing to change plans. The active payment method is displayed on the dashboard for reference.
Issue
subscription.updated webhook fires for multiple states — processing all of them causes incorrect DB updates.
Fix
Only events with status === 'active' are processed. All other statuses are stored in the event log for auditing but skipped for DB mutations.
Issue
Manually deriving current_start / current_end for billing cycles is imprecise and error-prone.
Fix
The subscription.charged webhook provides exact timestamps from Razorpay. These are stored directly on the user record — no manual date arithmetic required.
Issue
User can upload files even when subscription is in paused, in_grace, or past_due state.
Fix
Upload endpoint validates subscription state before accepting any file. States that block uploads: paused, in_grace, past_due. Edge cases that allow uploads: cancelled with cancelAtPeriodEnd=true (still within paid period) and awaiting_activation.

Setup Guide

1
Clone & install dependencies
git clone the repo, then run npm install in both /backend and /frontend.
2
Configure environment variables
Copy .env.example to .env. Fill in MongoDB URI, AWS credentials (S3_BUCKET, CLOUDFRONT_DOMAIN), Razorpay keys (RAZORPAY_KEY_ID, RAZORPAY_SECRET), plan IDs (PLAN_BASIC_MONTHLY etc.), and Gmail OAuth2 credentials.
3
Set up AWS S3 + CloudFront
Create an S3 bucket with Intelligent Tiering enabled. Create a CloudFront distribution pointing at the bucket. Enable cache invalidation permissions for the IAM role.
4
Configure Razorpay plans
Create 6 plans in the Razorpay Dashboard (Basic/Standard/Pro × Monthly/Yearly). Copy the plan IDs into environment variables. Register the webhook endpoint (/api/webhooks/razorpay) in the Razorpay console.
5
Set up Gmail OAuth2
Create a Google Cloud project, enable Gmail API, create OAuth credentials. Visit developers.google.com/oauthplayground, authorize Gmail scopes, and generate a refresh token. Add GMAIL_CLIENT_ID, GMAIL_CLIENT_SECRET, GMAIL_REFRESH_TOKEN to .env.
6
Run the app
npm run dev in backend (starts Express + cron jobs), npm start in frontend (starts React). Backend defaults to :5000, frontend to :3000.
